# StudyStack

**StudyStack** — это ИИ-генератор длинных академических текстов. Построен на микросервисной LLM-архитектуре с использованием LangGraph и LangChain. Основная задача — автоматическая генерация объемных, структурированных и фактологически точных документов (до 15-30 страниц), обходя классические ограничения LLM.

Доступен через Telegram-бота с потоковым выводом статуса графа, а также через CLI-интерфейс.

---

## Решаемые инженерные проблемы

При разработке LLM-решений для генерации длинных текстов возникают фундаментальные ограничения:
- **Ограничения контекста (Context Window Limits) и вывода (Max Output Tokens):** Модели не способны сгенерировать 15+ страниц связного текста за один вызов.
- **Lost in the Middle:** При передаче большого объема собранных фактов в контекст, модели теряют информацию из середины промпта.
- **Галлюцинации:** Пытаясь заполнить объем при недостатке контекста, LLM начинают выдумывать факты, статистику и цитаты.
- **Рассинхронизация при распараллеливании:** Независимо генерируемые главы теряют логическую связность и дублируют выводы.

**StudyStack** решает эти проблемы через распределенный Map-Reduce конвейер: строгая сегментация контекста, изоляция данных между агентами и контролируемая параллельная генерация.

---

## Технологический стек

- **Core:** Python 3, `asyncio` (конкурентное выполнение узлов графа)
- **Оркестрация LLM:** LangGraph, LangChain (включая сабграфы и внутреннюю рекурсию)
- **Схемы и валидация:** Pydantic (строгий Structured Output, State-схемы), Pydantic Settings (`.env`)
- **Сетевое взаимодействие и парсинг:** `httpx`, DuckDuckGo Search API (`DDGS`), **`r.jina.ai`** (прямой парсинг веб-страниц в Markdown)
- **Пользовательские интерфейсы:** Aiogram 3 (Telegram-бот с FSM), CLI-клиент с потоковым трекингом графа
- **LLM-провайдеры:** Поддержка OpenAI API (совместимость с локальными моделями) с архитектурным разделением на "тяжелые" (reasoning) и "дешевые" (компрессия/linter) модели.

## Архитектура системы

Пайплайн базируется на концепции графа состояний (`StateGraph`), где каждый узел выполняет узкоспециализированную задачу с жестко лимитированным контекстом.

### 1. Macro Planner (Планирование)
На основе темы, академического уровня и заданного объема, агент-планировщик формирует структуру будущего документа. Он гарантирует минимум 5 глав, включая Введение, Заключение и Библиографию, динамически распределяя требуемое количество страниц (`page_count`) по разделам. Результат — массив `macro_plan`.

### 2. Smart Search & Map-Scoring (Fan-Out / Fan-In)
- **Поиск (Source Finder):** Агент-ресерчер реализован как внутренний сабграф (`agent` -> `tools` -> `formatter`). Он использует кастомный тул `search_web` (на базе DDGS с обходом лимитов и Exponential Backoff), циклично собирая ровно `N + 7` источников. В случае превышения лимита вызовов (`GraphRecursionError`) применяется graceful fallback с уже собранными данными. Ссылки попадают в `source_registry`.
- **Анализ и распределение (Source Analyzer):** Обработка данных распараллеливается через `Send` (Fan-Out). Каждая ссылка обрабатывается независимым воркером:
  - Статья скачивается и конвертируется в Markdown через **`r.jina.ai`** (с пулом на 10 попыток и Backoff).
  - Выжимка формируется "дешевой" LLM (экстракция фактов и цитат).
  - Воркер классифицирует факты, привязывая их строго к одной, самой подходящей главе из `macro_plan` (`plan_indexes`).
- **Агрегация (Reducer):** Изолированные факты сливаются в функцию-редюсер `merge_classification_results`. Каждая глава получает уникальный, компактный пул проверенной информации.

### 3. Iterative Scriptor (Логический каркас)
Агент-Сценарист работает в цикле по главам, имея доступ только к фактам текущей главы. Для каждой из них формируются `core_thesis` (основной тезис) и `bridge_to_next` (логический мост). 
*Особенность:* Введение и Заключение проектируются в последнюю очередь. Сценарист получает на вход "скелет" всего остального документа, что исключает галлюцинации и позволяет написать точное вступление.

### 4. Parallel Expand (Генерация текста)
Граф распадается на параллельные ветки (сабграф `Chapter Subgraph`) по количеству глав:
- **Writer:** Агент-Писатель получает только свои факты, тезис и переходы. Он динамически рассчитывает нужный объем (`page_count * 400` слов). Библиография обрабатывается по отдельному системному промпту.
- **Reviewer:** Внутри каждой ветки работает проверяющий агент. Он сверяет написанный черновик с фактами из пула. Если найдены галлюцинации или отклонения, глава уходит на цикл переписывания (`needs_rewrite`). Лимит — 3 итерации.
- **Linter:** Успешно прошедшие ревью главы форматируются Markdown-линтером (LLM), который восстанавливает иерархию заголовков (`##`, `###`) и очищает артефакты генерации.

### 5. Final Assembly
Функция-редюсер (`merge_drafts_reducer`) собирает финальные тексты из всех веток, а узел `final_concatenator` склеивает их в итоговый Markdown-документ. Бот отдает готовый файл пользователю в Telegram напрямую из памяти (`BufferedInputFile`).

## Запуск проекта

1. Клонируйте репозиторий:
   ```bash
   git clone https://github.com/AizeRs/StudyStack.git
   cd StudyStack
   ```

2. Настройка виртуального окружения:
   ```bash
   python -m venv venv
   source venv/bin/activate  # Для Windows: venv\Scripts\activate
   pip install -r requirements.txt
   ```

3. Конфигурация:
   - Скопируйте `.env.example` в `.env`
   - Заполните необходимые ключи (токен Telegram бота, ключи для LLM)
   - Поддерживаются локальные модели (`USE_LOCAL_LLM=True`)

4. Запуск Telegram-бота:
   ```bash
   python app/tg_bot/bot.py
   ```
   *(Также доступна CLI-версия через `python main.py`)*
